# Audit de mise à jour — BDD-DOC (périmètre `src` uniquement)

_Date de l'audit : 24 avril 2026_

## Objectif

Refaire un audit rapide et factuel du dépôt avec un périmètre strictement limité à `src/` (sans analyse de `vba_import/`).

## Méthodologie

- Relecture ciblée des modules sensibles dans `src/` :
  - `zDocConstance.bas`
  - `zDocGitSync.bas`
  - `zDocDeveloppeurOnOFF.bas`
  - `zDocImportBDD.bas`
  - `BoutonZoom.bas`
- Mesures sur patterns à risque dans `src/` uniquement :
  - `On Error Resume Next`
  - `CreateObject(...)`
  - `Application.Run`
  - `SendKeys`
- Vérification de la discipline `Option Explicit` sur les modules texte exportés de `src/`.

## Résultats clés

## ✅ Points positifs observés dans `src`

1. **Gestion du mot de passe développeur externalisée** : la valeur est lue via `BDD_DOC_DEV_PASSWORD` et n'est pas stockée en clair dans le code.
2. **Configuration repo GitSync portable** : le chemin repo est résolu via `BDD_DOC_REPO_PATH` puis fallback `ThisWorkbook.Path`.
3. **Base modulaire globalement structurée** : séparation logique par domaines (import, UI, sécurité, ouverture docs, performance).

## ⚠️ Risques encore ouverts (priorisés)

### R1 — Gestion d'erreur permissive encore trop présente (Élevé)

- `On Error Resume Next` reste fréquent dans `src` (38 occurrences mesurées).
- Impact : erreurs silencieuses, détection tardive des anomalies en production.
- Recommandation : limiter `Resume Next` à des blocs ultra-courts, puis contrôle explicite + journalisation.

### R2 — Surface d'exécution dynamique et dépendance environnement Office/Windows (Élevé)

- Présence de `CreateObject` (18 occurrences), `Application.Run` (6 occurrences), `SendKeys` (1 occurrence).
- Impact : sensibilité au contexte poste utilisateur (droits VBProject, composants installés, état UI active).
- Recommandation : encapsuler les appels dynamiques et renforcer les prérequis/guardrails au point d'entrée.

### R3 — Discipline VBA incomplète sur `Option Explicit` (Moyen)

- 1 module sans `Option Explicit` détecté : `src/BoutonZoom.bas`.
- Impact : risque d'erreurs liées aux variables implicites.
- Recommandation : ajouter `Option Explicit` immédiatement (quick win).

### R4 — Dépendances UI non déterministes (Moyen)

- Certaines routines s'appuient sur `ActiveWindow` et l'état visuel courant.
- Impact : comportements variables selon multi-écran, session distante, fenêtre inactive.
- Recommandation : réduire l'implicite UI quand une alternative déterministe existe.

## Recommandations concrètes (ordre d'exécution)

1. Ajouter `Option Explicit` dans `src/BoutonZoom.bas`.
2. Lancer une passe "durcissement erreurs" sur les modules les plus sensibles (`zDocImportBDD`, `zDocGitSync`, `zDocDeveloppeurOnOFF`).
3. Encadrer `CreateObject`/`Application.Run`/`SendKeys` avec checks de prérequis + messages d'erreur actionnables.
4. Ajouter un check local automatique (pré-commit ou CI) sur :
   - présence de `Option Explicit` dans tous les modules texte de `src/`,
   - seuil maximal d'occurrences `On Error Resume Next` (à réduire progressivement).

## Synthèse exécutive

Sur le périmètre `src/`, le projet est en meilleure posture qu'avant (secret et portabilité mieux gérés) mais conserve un risque de robustesse principalement lié à la gestion d'erreur permissive et aux mécanismes d'exécution dynamiques. Une itération ciblée sur ces points apportera un gain de fiabilité rapide sans refonte lourde.
